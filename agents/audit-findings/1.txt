4.1.1. Subscriber Dos On Burn
Unit Name: _burn
Location: contracts/evm/lib/v4-periphery/src/PositionManager.sol, _burn function;
contracts/evm/lib/v4-periphery/src/base/Notifier.sol, _removeSubscriberAndNotifyBurn and
_call functions
Description: When a position has a subscriber ( info.hasSubscriber() is true), the _burn
function calls _removeSubscriberAndNotifyBurn , which invokes the subscriber's notifyBurn
callback via _call() . The _call function uses a low-level call that forwards ALL remaining
gas to the subscriber. In the Fiet Protocol, the subscriber is DirectLPDeltaResolver , which calls
factory.afterModifyLiquidity(poolKey) , which in turn calls CoreHook.settleHookDeltasToPot(key) . If the
factory call reverts (due to bounds check failure via Bounds.isEndpoint , the factory being
unregistered, or the factory contract being malicious), the _call returns false, and
_removeSubscriberAndNotifyBurn executes bubbleUpAndRevertWith , reverting the entire burn
transaction. This creates a DoS vector: if the factory associated with a pool becomes
unresponsive, is replaced, or if the subscriber's bounds are revoked, users cannot burn
their positions. Additionally, since the subscriber deletion ( delete subscriber[tokenId] ) occurs
before the external call but the revert rolls back all state changes, the subscriber cannot be
removed to unblock the burn — the user is stuck until the factory issue is resolved.
User Verdict: True Positive
