Global one‑use settlement proof‑by‑hash in VRLSettlementObserver (lane‑scoped verification) causes frontrunnable denial of grace extension and potential position seizure

VRLSettlementObserver verifies settlement proofs only against (poolId, tokenIndex) but consumes them globally by hash on first use. An attacker with a dust position in the same lane can copy a victim’s proof from the mempool and consume it first, causing the victim’s grace extension to revert. With the PR’s checkpoint semantics (inherited openSince and lane‑local extension reset on toggles), this can immediately enable seizure and loss of the victim’s liquidity.

The settlement observer computes proofHash = hash(settlementProof) and, upon successful verification against (poolId, tokenIndex), sets usedProofHashes[proofHash] = true. Subsequent attempts with the same bytes revert as InvalidProof. Because the verifier attests only the lane (poolId, tokenIndex) and no position identifier is provided on-chain, identical proof bytes can be valid for multiple positions in the same lane. VTSOrchestrator.extendGracePeriod is callable by bound endpoints (e.g., MMPositionManager). An attacker who holds a dust position in the same lane can front-run the victim by submitting an extend call for their own position with the victim’s proof bytes, thereby consuming the proof globally. The victim’s later extend attempt then reverts with InvalidProof. The PR’s checkpoint semantics increase impact: on lane toggles, lane-local grace extensions reset to zero while a canonical openSince is inherited across lane-composition changes, so a position may already be near or past base grace when extension is denied. After grace elapses, anyone can initiate seizure, causing a direct reduction of the victim’s liquidity units.
