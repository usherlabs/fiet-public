4.2.7. Stuck Eth In Payable Erc20 Wrap
Unit Name: LiquidityHub
Location: contracts/evm/src/LiquidityHub.sol — wrapTo and wrap external payable
functions, _wrap internal function
Description: The wrapTo(address lcc, address to, uint256 amount) , wrapTo(address underlying, bytes32
marketId, address to, uint256 amount) , wrap(address lcc, uint256 amount) , and wrap(address underlying,
bytes32 marketId, uint256 amount) functions are all marked payable . In the _wrap internal
function, msg.value is only checked when the underlying is a native asset ( underlying ==
address(0) ): if (msg.value != amount) revert . For ERC20-backed LCCs where underlying !=
address(0) , there is no check that msg.value == 0.
If a user accidentally sends ETH along with an ERC20 wrap call, the ETH is deposited into
the LiquidityHub contract. The contract's receive() function only accepts ETH from validated
MarketVault contracts (via _assertValidEthSender() ), so there is no direct recovery mechanism
for the stuck ETH. The stuck ETH inflates address(this).balance , which can aﬀect the
confirmTakeBalanceInvariant check for native-ETH-backed LCCs by making the actual balance
appear higher than the tracked reserve, potentially allowing an issuer to increment native
reserves beyond what was legitimately deposited from market operations.
User Verdict: True Positive
