````markdown
## Vulnerability Details

### Finding: Silent Confirm Take in MarketVault Recipient == LiquidityHub Path Causes Reactive Settlements to Stall

## Description

When `MarketFactory.useMarketLiquidity` routes withdrawals to `LiquidityHub`, `MarketVault` calls `LiquidityHub.confirmTake` with `shouldEmit=false`. This increases pooled market-derived reserves without emitting `LiquidityAvailable`, so `HubRSC` won't dispatch external settlements until another liquidity signal occurs.

### Detailed Explanation

The `MarketFactory.useMarketLiquidity` flow triggers `IMarketVault.tryModifyLiquiditiesWithRecipient` with `recipient=LiquidityHub`. In `MarketVault._finaliseModifyLiquidities`, when `recipient==LiquidityHub`, the code calls:

```solidity
liquidityHub.confirmTake(lcc, used, false)
```
````

**The Problem:**

- `LiquidityHub.confirmTake` increments the shared `reserveOfUnderlying.marketDerived`
- However, it only emits `LiquidityAvailable` when `shouldEmit=true`
- `HubRSC`, which schedules batched settlements, only dispatches on:
  - `LiquidityAvailable` events
  - `MoreLiquidityAvailable` events (emitted after a previous dispatch with leftovers)
- `HubRSC` does **NOT** dispatch upon `SettlementReported` alone

**Result:**
When `confirmTake(false)` increases pooled reserves without an immediate external payout (notably in wrap-with residual unwrapping, where `queueTo=Hub` and Hub's own queue burns do not decrement reserve), external queues that are now serviceable will not be dispatched by the reactive layer until:

- A future liquidity signal is emitted, OR
- Someone manually calls `processSettlementFor`

This behavior change is introduced in this PR via the new/changed `confirmTake(…, shouldEmit)` semantics and `MarketVault`'s use of `shouldEmit=false` on the `useMarketLiquidity` path.

## Mitigation Guide

The code comments suggest the following mitigations:

1. **In MarketVault.sol**: Consider passing `shouldEmit=true` to re-enable reactive scheduling for market-driven takes, paired with gating in `LiquidityHub.confirmTake` to only emit when external queues exist.

2. **In LiquidityHub.sol**: After `wrapWithContext`, emit `LiquidityAvailable` for both LCCs, sized to per-LCC queued totals and shared underlying reserve:
   ```solidity
   uint256 underlying = s.lccToUnderlying[lcc];
   uint256 available = s.reserveOfUnderlying[underlying].marketDerived;
   uint256 amtL = Math.min(available, s.totalQueued[lcc]);
   uint256 amtW = Math.min(available > amtL ? (available - amtL) : 0, s.totalQueued[withLCC]);
   if (amtL > 0) emit LiquidityAvailable(lcc, underlying, amtL, s.lccToMarket[lcc].id);
   if (amtW > 0) emit LiquidityAvailable(withLCC, underlying, amtW, s.lccToMarket[withLCC].id);
   ```
