# Security Vulnerability Report: EIP-1153-Dependent Reentrancy Guard in LiquidityHub

## Overview
The `LiquidityHub` contract now inherits OpenZeppelin’s `ReentrancyGuardTransient` and applies the `nonReentrant` modifier to core entrypoints. This introduces a **functional Denial-of-Service (DoS)** on chains or environments that do not support **EIP-1153 (Cancun)**, as the transient storage operations (`TLOAD`/`TSTORE`) required by the guard will revert.

This is a **PR-introduced compatibility regression** specific to `LiquidityHub`. While `VTSOrchestrator` and `MarketVault` also inherit the guard, they already depended on Uniswap v4’s transient storage usage, so their chain requirements remain unchanged.

---

## Vulnerability Details

### Severity
- **Severity**: Medium
- **Likelihood**: Low
- **Impact**: High

### Affected Contracts
- `LiquidityHub.sol`
- `MarketVault.sol`
- `VTSOrchestrator.sol`

### Affected Functions
All core entrypoints in `LiquidityHub.sol` annotated with `nonReentrant`:
- `wrap`, `wrapTo`
- `unwrap`, `unwrapTo`
- `issue`, `cancel`, `cancelWithQueue`
- `queueForTransferRecipient`
- `planCancel`, `planCancelWithQueue`
- `prepareSettle`, `processSettlementFor`

---

## Technical Description

The vulnerability arises because `ReentrancyGuardTransient` relies on **EIP-1153 transient storage**, which is not available on all EVM chains or local development environments. When a contract calls any of the `nonReentrant`-protected functions on a non-EIP-1153 chain, the transaction reverts immediately at the modifier level, preventing any core operations like wrapping, unwrapping, or settlement.

No fallback mechanism (e.g., storage-based reentrancy guard) or feature detection is implemented to handle non-compliant chains.

## Related Findings

### Transient nonReentrant flag stickiness in MarketVault liquidity entrypoints
- **Severity**: Low
- **Description**: `MarketVault` now uses `ReentrancyGuardTransient` on liquidity entrypoints. Due to EIP-1153 dependency, same-transaction calls may fail if transient storage state is not properly cleared.

---

## Recommendation

1. **Add EIP-1153 Feature Detection** — Check chain compatibility before using `ReentrancyGuardTransient`.
2. **Implement Fallback Guard** — Use a storage-based reentrancy guard (e.g., `ReentrancyGuard`) for non-EIP-1153 chains.
3. **Document Chain Requirements** — Clearly state in documentation that `LiquidityHub` requires EIP-1153 support.
4. **Test on Non-Cancun Chains** — Add test coverage for pre-Cancun environments to catch regressions.
