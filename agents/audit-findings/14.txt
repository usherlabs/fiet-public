# Security Vulnerability Report: Per-(lcc,from,to) Transient Planned-Cancel Without Transfer Linkage

## Overview
The `VTSPositionLib` and `LiquidityHub` contracts implement a **planned-cancel-with-queue** mechanism using transient storage slots keyed by `(lcc, from, to)`. This design **lacks per-transfer linkage**, leading to **lost or misapplied cancellations** and **stuck/misattributed settlements** during batched liquidity decreases.

This is a **medium-severity design flaw** introduced in PR #135, affecting settlement integrity and accounting correctness — not direct principal loss, due to auto-forwarding and settlement serviceability checks.

---

## Vulnerability Details

### Severity
- **Severity**: Medium
- **Likelihood**: Medium
- **Impact**: Medium

### Affected Contracts
- `VTSPositionLib.sol`
- `LiquidityHub.sol`

### Affected Functions
- `VTSPositionLib._handleLiquidityDecrease`
- `LiquidityHub.planCancelWithQueue`
- `LCC._afterTransfer` → `LiquidityHub.executePlannedCancel`

---

## Technical Description

The system schedules cancellation plans via `LiquidityHub.planCancelWithQueue` using `principalDelta` (converted to absolute value), **without verifying whether an actual `PoolManager → MMPM` transfer occurs** in the same call.

This creates three failure modes:
1. **Stale Plans**: Plans written when `callerDelta.amountX() ≤ 0` (no take) remain unconsumed.
2. **Overwritten Plans**: Later actions in the same transaction overwrite prior plans for the same `(lcc, from, to)` key.
3. **Misapplied Consumption**: Later transfers consume stale plans not intended for them, causing misattribution.

Because cancellation is not tied to actual transfer outcomes, **settlement queues become inconsistent**, leading to:
- Lost cancellations (under-application)
- Stuck or misattributed settlements

### VTSPositionLib.sol — Line 1473–1491
```solidity
function _handleLiquidityDecrease(
    Context memory ctx,
    address owner,
    PoolKey calldata poolKey,
    BalanceDelta principalDelta,
    BalanceDelta requiredSettlementDelta,
    address queueRecipient
) internal returns (BalanceDelta settleableDelta) {
    if (LiquidityUtils.isZeroDelta(principalDelta)) {
        return BalanceDelta.wrap(0);
    }

    // Only plan for positive principal, and cap by actual take amounts
    uint256 principalAmount0 = principalDelta.amount0() > 0 ? uint256(int256(principalDelta.amount0())) : 0;
    uint256 principalAmount1 = principalDelta.amount1() > 0 ? uint256(int256(principalDelta.amount1())) : 0;
    uint256 takeAmount0 = callerDelta.amount0() > 0 ? uint256(int256(callerDelta.amount0())) : 0;
    uint256 takeAmount1 = callerDelta.amount1() > 0 ? uint256(int256(callerDelta.amount1())) : 0;

    if (principalAmount0 > takeAmount0) principalAmount0 = takeAmount0;
    if (principalAmount1 > takeAmount1) principalAmount1 = takeAmount1;
```

---

## Exploitation Scenario

1. User initiates a batched liquidity decrease with multiple tokens.
2. For token A, `principalDelta > 0` but `callerDelta ≤ 0` → plan is scheduled but no transfer occurs.
3. Later in same tx, token B triggers a transfer → consumes stale plan for token A.
4. Result: Token A’s cancellation is lost; Token B’s settlement is misattributed.

---

## Recommendation

1. **Tie Plans to Transfer Events** — Only schedule cancellation plans when an actual `PoolManager → MMPM` transfer is confirmed in the same call.
2. **Use Per-Transfer Keys** — Include transaction nonce or transfer ID in transient slot key to prevent overwrites.
3. **Add Validation** — In `executePlannedCancel`, verify the plan matches the current transfer context.
4. **Audit Batched Operations** — Ensure all batched liquidity changes properly sequence plan creation and consumption.

# Finding 14 Triage: Planned-cancel transient-slot overwrite risk needs sequencing review

## Verdict
Not a confirmed live vulnerability on the current MM decrease call graph.

Treat this as a sequencing-sensitive design consideration and documented invariant, not implementation work, unless a
new flow is shown to break immediate consumption.

## Summary
`LiquidityHub.planCancel(...)` and `planCancelWithQueue(...)` key transient intent by `(lcc, from, to)`, so abstract
same-key overwrite is possible in principle.

However, the current MM decrease path does not appear to leave these plans sitting across unrelated intermediate logic:

1. `VTSPositionLib._handleLiquidityDecrease(...)` stages the plan during `PoolManager.modifyLiquidity(...)` hook
   execution, explicitly because the LCC has not yet been transferred to `MMPM`.
2. `PositionManagerImpl._modifySyntheticLiquidity(...)` returns from `modifyLiquidity(...)` and immediately processes
   positive deltas.
3. `PositionManagerImpl._takePositiveDeltasAndHandleLcc(...)` performs the matching `PoolManager -> MMPM` take for the
   affected `lcc`.
4. `PositionManagerImpl._handleLccBalanceIncrease(...)` forwards non-fee LCC onward, where the transfer path triggers
   `LiquidityHub.executePlannedCancel(...)`.

This means the current design relies on an adjacency invariant:

- the plan is staged and then consumed in the same logical path and transaction,
- before another same-key plan can be restaged.

## Why the original concern is still worth documenting
The `(lcc, from, to)` key is still coarse.

If a future refactor introduces any of the following before consumption, the original overwrite concern becomes real:

- deferred LCC takes after `modifyLiquidity(...)`,
- nested batching that can restage the same path,
- retry/fallback logic between planning and transfer,
- an alternate transfer destination for the same logical decrease,
- multiple same-key planned cancels before the first matching transfer executes.

So the concern is best framed as:

- not a demonstrated exploit on the present call graph,
- but a brittle sequencing assumption that must remain explicit in code review and tests.

## Requested handling

1. Document the invariant in code and protocol notes.
2. Preserve the immediate `plan -> matching transfer -> consume` adjacency in MM decrease flows.
3. Re-open as an implementation issue only if a reachable same-key re-plan before consumption is demonstrated, or if a
   new flow weakens the current sequencing.

## Suggested severity
Informational / needs human confirmation, pending any concrete counterexample to the current immediate-consumption path.
