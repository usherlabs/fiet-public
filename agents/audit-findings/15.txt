# Security Vulnerability Report: DICE Residual Flush on First Deficit in VTSCommitLib/VTSPositionLib Causes Concentrated Burn of Victim’s Unburned Fees

## Overview
When a pool has no outstanding deficit principal, coverage increments are deferred to a residual. The **first position to later incur any deficit triggers a residual flush and immediate coverage application**, and the fee-burn scales by the new outflow window that just opened. An attacker can time this boundary to **slash a victim’s unburned fees (on the opposite token) up to coverageFeeShare% in one shot**.

This is a **medium-severity logic flaw** introduced by PR #135’s DICE model changes, enabling **griefing attacks** via public settlement functions.

- `VTSPositionLib._applyCoverageBurn`

## Technical Description

The PR’s DICE model defers coverage increments to `coverageResidualDICE` when `paPool.totalDeficitPrincipal == 0`. The first time any position materializes deficit principal, `VTSPositionLib._settlePositionDeficitGrowth` calls `_flushCoverageResidualIfNeeded`, which:

1. Advances `coveragePerDeficitIndexX128` by `residual * Q128 / totalDeficitPrincipal`
2. Clears the residual

In the same `settlePositionGrowths` pass, `_settleDeficitIndexedCoverageUsage` applies coverage to that position (`cov = deficitPrincipal * deltaIndex / Q128`), then `_applyCoverageBurn` computes:

```solidity
feesBurn = fees_since_last_snap * (burnBase / ofDelta) * bps
```

Where:
- `ofDelta = cumulativeOutflows - outflowsAtFeeSnap` (new outflow window)
- `burnBase = cov < d ? cov : d` (coverage or deficit, whichever is smaller)

Because the **first tiny deficit opens a new outflow window** (`ofDelta ≈ d`), the ratio `burnBase / ofDelta ≈ 1`, so the victim’s unburned fees are slashed up to `coverageFeeShare%`.

Since `settlePositionGrowths` is **public**, an attacker can force this timing on a chosen victim.

## Code Context

### VTSPositionLib.sol — Lines 757–777
```solidity
uint256 indexLast = pa.coverageIndexLastX128.get(tokenIndex);

if (indexNow != indexLast) {
    pa.coverageIndexLastX128.set(tokenIndex, indexNow);
}

uint256 deltaIndex = indexNow - indexLast;

if (deltaIndex > 0) {
    uint256 d = pa.cumulativeDeficit.get(tokenIndex);
    uint256 cov = FullMath.mulDiv(d, deltaIndex, FixedPoint128.Q128);

    if (cov > 0) {
        // Gate first-deficit spikes: require a prior outflow window larger than the current exercise
        uint256 cf = pa.cumulativeOutflows.get(tokenIndex);
        uint256 snap = pa.outflowsAtFeeSnap.get(tokenIndex);
        uint256 ofDelta = cf >= snap ? (cf - snap) : 0;
        uint256 burnBase = cov < d ? cov : d;

        if (ofDelta > burnBase) {
            _applyCoverageBurn(s, poolManager, positionId, poolId, tokenIndex, cov, liq);
            pa.coverageIndexLastX128.set(tokenIndex, indexNow);
        }
    } else if (indexNow != indexLast) {
        pa.coverageIndexLastX128.set(tokenIndex, indexNow);
    }
}
```

## Exploitation Scenario

1. Victim holds a position with unburned fees on token A.
2. Attacker opens a tiny deficit position on token B (same pool).
3. Attacker calls `settlePositionGrowths` on victim’s position.
4. System flushes residual, applies coverage, and burns victim’s fees on token A up to `coverageFeeShare%`.
5. Attacker repeats to drain victim’s unburned fees.

---

## Recommendation

1. **Delay Burn Until Next Window** — Do not apply coverage burn on the same transaction that opens a new outflow window.
2. **Require Minimum Window Size** — Enforce `ofDelta > minThreshold` before applying burn.
3. **Rate-Limit Settlements** — Add cooldown or gas cost to public settlement functions.
4. **Add Access Control** — Restrict `settlePositionGrowths` to internal or trusted roles.
