Ignored origin check in BatchProcessSettlement.processSettlements causes receiver-funded callback griefing

The new destination receiver’s processSettlements only gates on the shared callback proxy and ignores the callbackOrigin (sender RVM ID), allowing any proxied Reactive callback to trigger batches of LiquidityHub.processSettlementFor at the receiver’s expense. This enables cost/liveness griefing via accumulated proxy debt and forced payment through coverDebt().

This PR introduces contracts/reactive/src/dest/BatchProcessSettlement.sol with an external entrypoint processSettlements(address callbackOrigin, ...). The function enforces only authorizedSenderOnly (the protocol-chain callback proxy) and explicitly ignores callbackOrigin, which carries the upstream Reactive contract’s RVM ID. As a result, any Reactive contract able to emit a callback through the shared proxy can invoke this receiver and cause it to execute batches of LiquidityHub.processSettlementFor. LiquidityHub remains permissionless and enforces its own invariants, so no unauthorized fund movement occurs. However, the receiver (wired to the proxy via AbstractCallback/AbstractPayer) accrues callback debt with the proxy for these invocations. Anyone can then call coverDebt() on the receiver to settle that debt from the receiver’s balance (prefunded at deploy), draining funds and/or degrading liveness if debt remains unpaid. Other components in this PR (e.g., HubCallback methods validate origin RVM IDs), but the receiver does not, creating an inconsistent and broader external trigger surface introduced by this PR.
