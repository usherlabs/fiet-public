openSince inheritance across lanes in RFS checkpointing allows premature seizure due to stale checkpoint

A PR change makes a newly opened RFS lane inherit the other lane’s prior openSince. Because checkpoint writes are discrete and permissionless, attackers can force inheritance from stale state (skipping an unrecorded fully-closed episode), making the newly opened lane immediately grace-eligible and enabling premature seizure.

The PR modified RFSCheckpointLibrary.mark so that when one lane opens while the other was previously open, the newly opened lane’s openSince is inherited from the other lane’s prior openSince (inheritedOpenSince1 -> openSince1 assignment). Lane-local grace extensions are reset for the lane that toggles. Seizability in CheckpointLibrary.isSeizable is lane-scoped and aggregated via OR using (openSince + baseGrace + extension). VTSOrchestrator.checkpoint is permissionless and writes the checkpoint state; VTSOrchestrator.settlePositionGrowths is also permissionless (when unpaused) and updates growth/deficit/inflow but does not mark the checkpoint. onSeize relies on the stored checkpoint unless a stored commitment deficit exists (only then it recomputes/marks before checking). An attacker can exploit this by allowing a real fully-closed episode (00) to occur without being written, and later calling checkpoint exactly when the lane composition flips (e.g., 01->10 or 01->11). In mark, the newly opened lane inherits the old openSince from the other lane, making its grace window appear elapsed immediately. As a result, onSeize deems the position seizable and the attacker can seize liquidity prematurely. Pre-PR behavior set openSince on each lane’s open to block.timestamp, preventing this premature eligibility.
