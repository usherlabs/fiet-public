4.1.2. Double Credit Same Balance
Unit Name: VTSCurrencyDelta
Location: contracts/evm/src/modules/VTSCurrencyDelta.sol — sync() and syncPair() functions, and
contracts/evm/src/libraries/DynamicCurrencyDelta.sol — syncBalanceAsCredit() function
Description: The syncBalanceAsCredit() function in DynamicCurrencyDelta reads the owner 's
token balance via currency.balanceOf(owner) and credits the diﬀerence between that balance
and the target 's current delta. Critically, the function does not track or debit the owner's
balance after syncing — it only reads it. This allows the same owner balance to be synced
to credit multiple diﬀerent target addresses.
Attack scenario using sync() or syncPair() (which delegate to syncBalanceAsCredit ):
1. Call sync(factory, currency, owner=contractA, target=userA) — contractA has balance
1000, userA has delta 0, so userA gets credited 1000.
2. Call sync(factory, currency, owner=contractA, target=userB) — contractA still has balance
1000 (never decremented), userB has delta 0, so userB also gets credited 1000.
Total delta credits created: 2000, backed by only 1000 in actual balance. This directly
violates the project invariant: "DynamicCurrencyDelta must not double-credit the same
balance when tracking currency deltas for the same token in the same transaction context"
and "Dynamic currency deltas must net to zero at the end of each PoolManager unlock
callback: total credits must equal total debits across all currencies for proper settlement." A
protocol-bound caller can trigger this by making two sync calls with diﬀerent target
parameters but the same owner , creating unbacked credits that can then be settled for real
tokens.
User Verdict: True Positive
