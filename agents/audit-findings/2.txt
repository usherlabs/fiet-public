Inherited openSince on second-lane opens (01->11/10->11) in RFSCheckpointLibrary.mark causes premature position seizure

A PR change makes a newly opened RFS lane inherit the opposite lane’s prior openSince when that opposite lane is already open, including on 01->11/10->11 transitions. Since seizability checks are OR-based per-lane with lane-local grace, an attacker can create a fresh shortfall on the short-grace lane, permissionlessly checkpoint to inherit an older openSince, immediately pass grace on that fresh lane, and seize the position while funding deposits largely on the opposite lane with higher exposure.

contracts/evm/src/types/Checkpoint.sol (RFSCheckpointLibrary.mark) now sets a newly opened lane’s openSince to the opposite lane’s previous openSince if the opposite lane was already open. This applies to opening a second lane (01->11 or 10->11), not just pure rotations. Lane-specific grace extensions reset on lane toggles. Seizability (contracts/evm/src/libraries/Checkpoint.sol) is OR-based across lanes using lane-local grace windows; there is no minimum-age gating on the normal RFS path. Attackers can (a) induce a fresh RFS requirement on the short-grace lane via swaps, (b) call VTSOrchestrator.checkpoint() (permissionless) to mark lanes so the fresh lane inherits the older openSince, immediately satisfying its grace, then (c) call the MMPositionManager seizure path (third-party seizers are allowed for active positions). During settlement, VTSPositionLib._calcSeizure adds exposure from both lanes, so the attacker can deposit predominantly on the opposite lane with high RFS to seize significant liquidity. onSeize uses the stored checkpoint for normal RFS (no recompute unless a stored commitment deficit exists), so the attacker-controlled checkpointed state is honored. Before this PR, a newly opened lane would start openSince at block.timestamp, preventing immediate seizure on a fresh lane.
