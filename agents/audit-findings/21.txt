Cast-before-clamp inversion in ProxyHook _calcCoreSqrtPriceLimit causes incorrect core price-limit mapping

ProxyHook._calcCoreSqrtPriceLimit inverts a proxy sqrt price and casts to uint160 before clamping, which truncates for very small inputs and returns an incorrect mapped limit used by the core swap.

When the proxy and core swap directions are flipped, ProxyHook._calcCoreSqrtPriceLimit maps the proxy sqrtPriceLimitX96 to a core limit by computing inverted = uint160((1 << 192) / sqrtPriceLimitX96), then clamping to (MIN_SQRT_PRICE+1, MAX_SQRT_PRICE-1). For very small sqrtPriceLimitX96 (e.g., 1 or any value ≤ 2^32), the true 192-bit inversion exceeds 160 bits; casting to uint160 before clamping truncates the high bits. The clamp then runs on the truncated value rather than the full-precision inversion, potentially returning the wrong extreme (e.g., MIN+1 instead of MAX-1) or a midrange value that should have been saturated to MAX-1. Because ProxyHook cancels the proxy AMM swap by returning a BeforeSwapDelta that sets amountToSwap to 0, the proxy pool’s v4 bound checks are skipped, yet ProxyHook executes a core swap using the incorrectly mapped limit. Depending on direction and current price, this can allow a swap to proceed where it should have reverted, or cause unexpected reverts or unintended tighter bounds.
