Missing checkpoint refresh in VTSOrchestrator.onSeize for normal RFS path causes premature seizure (principal loss)

After a PR change, VTSOrchestrator.onSeize no longer refreshes RFS checkpoints for ordinary RFS-based seizures. Seizability is then evaluated against stale checkpoint timestamps, allowing a seizer to bypass the intended grace window and seize a victim’s liquidity early.

The PR modified VTSOrchestrator.onSeize so it it only refreshes checkpoints when a stored commitmentDeficit exists. On the normal RFS/grace path (no commitment deficit), onSeize now calls CheckpointLibrary.isSeizable directly on the stored checkpoint without first refreshing it. Because growth settlement is lazy, a position’s RFS can close and later reopen without the stored checkpoint being updated (settlePositionGrowths updates accounting but does not mark checkpoints). CheckpointLibrary.isSeizable relies solely on the stored checkpoint (openMask/openSince*/extensions) for the normal path. As a result, an old openSince from a previous RFS episode can remain in storage and appear to have passed grace, authorizing seizure even though the current reopened episode’s grace has not elapsed. The onMMSettle path repeats the same pre-settlement isSeizable check (still stale) and only afterwards settles growths and marks the checkpoint; it does not re-check grace after settlement. This enables premature, unintended seizure of a victim’s liquidity. The issue is introduced by the PR’s change to conditional checkpointing in onSeize for the normal RFS path.
