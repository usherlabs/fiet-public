4.1.3. Missing Access Control
Unit Name: VTSCurrencyDelta
Location: contracts/evm/src/modules/VTSCurrencyDelta.sol — take() function
Description: The take(Currency currency, address target, uint256 maxAmount) function in
VTSCurrencyDelta is public with no access control whatsoever. Every other state-modifying
function in the contract ( sync , syncPair, creditExact ) is protected by
_assertBoundFactoryCaller(factory) , which validates the caller is a protocol-bound address in a
registered factory. The take function lacks this guard entirely.
The underlying DynamicCurrencyDelta.take() reads a target's positive delta (credit), reduces it
by min(credit, maxAmount) via accountDelta(currency,
-SafeCast.toInt128(amountToTake), target) , and
returns the consumed amount. Any external caller can invoke vtsOrchestrator.take(currency,
victimAddress, 0) to zero out any address's entire credit delta for any currency. Multiple
protocol contracts (MMPositionManager, MMPositionActionsImpl, PositionManagerImpl,
PositionManagerEntrypoint) rely on calling vtsOrchestrator.take() to consume credits during
settlement — an unauthorized call to take before the intended consumer processes the
credit eﬀectively destroys the settlement claim.
Within a Uniswap V4 PoolManager unlock context where transient storage deltas are active,
any contract in the callback chain (including malicious contracts invoked via token transfer
hooks or LCC callbacks such as onSettleCallback or onTakeCallback ) can call take to drain
credits established by legitimate protocol operations. This violates the project invariant
that "Dynamic currency deltas must net to zero at the end of each PoolManager unlock
callback" because credits are destroyed without corresponding settlement. Additionally,
take has no nonReentrant modifier, so it is callable during reentrant paths from commitSignal,
onMMSettle , or other protected functions.
User Verdict: True Positive
