Paused remove-liquidity reconcile path omits MM side-effects in CoreHook/VTSOrchestrator causes principal LCC stuck in custodian and missing settlement deltas

When a market is paused, CoreHook routes remove-liquidity to VTSOrchestrator.reconcileAfterPausedRemove, which calls VTSPositionLib.reconcileAfterPausedRemove. This helper intentionally avoids MM bookkeeping (no LiquidityHub.planCancelWithQueue and no DynamicCurrencyDelta), but MMPositionManager still takes LCC from PoolManager and forwards principal to the queue custodian. Because no Hub queue is created, LiquidityHub.settleFromCustodian cannot release funds, leaving principal LCC stuck. DynamicCurrencyDelta is also not recorded, degrading later settlement flows.

This PR introduced a paused-only removal path: CoreHook.afterRemoveLiquidity, when a pool or the protocol is paused, calls VTSOrchestrator.reconcileAfterPausedRemove and returns immediately with a zero hook delta. VTSOrchestrator delegates to VTSPositionLib.reconcileAfterPausedRemove, which only tracks commitment changes, clamps settled amounts against updated commitment maxima, and mirrors liquidity; it explicitly avoids running the normal MM processing. In the unpaused path, VTSPositionLib._processMMOperations/_handleLiquidityDecrease stages LiquidityHub.planCancelWithQueue before the immediate post-modify PoolManager → MMPM transfer, and records DynamicCurrencyDelta for underlying settlement. In the new paused path, neither happens. MMPositionManager still performs the post-modify take, and LCC._afterTransfer attempts to consume a planned cancel, but none exists. The non-fee LCC increase is forwarded to the MMQueueCustodian (beneficiary = locker). Later, LiquidityHub.settleFromCustodian requires a Hub queue entry (settleQueue[lcc][locker] > 0) to settle; since no planned cancel created such a queue, it returns 0 and custody remains orphaned. There is no practical rescue path via existing role-gated entrypoints (issuers do not expose helpers; LiquidityHub is not an issuer; settleFromCustodian cannot proceed without a queue). Additionally, DynamicCurrencyDelta is not recorded for the paused decrease, which can mis-size or block subsequent settlement/seizure operations. This behavior is introduced by this PR because previously processPosition was guarded with notPoolPaused (so remove-liquidity under pause did not proceed), whereas now reconcileAfterPausedRemove runs under pause but omits required MM side-effects.
