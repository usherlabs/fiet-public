Stale RFS checkpoint usage in seizure validation allows premature seizure

The PR removed the pre-seizure checkpoint refresh for the normal RFS path, causing CheckpointLibrary.isSeizable to rely on a stale checkpoint (openSince/openMask) rather than live RFS state. This can allow seizure before a fresh grace period has elapsed after RFS recently re-opened.

Prior to the PR, onSeize always refreshed from live state (settled growths, optional commitment checks, computed getRFS, and marked the checkpoint) before evaluating seizability. The PR introduced VTSLifecycleLinkedLib.validateSeize and refactored onSeize/onMMSettle prechecks so that, when there is no stored commitmentDeficit, they no longer refresh and re-mark the checkpoint before calling CheckpointLibrary.isSeizable. Since public settlePositionGrowths and calcRFS can change live RFS status without marking the checkpoint, a position can be checkpointed 'open since T0', then live RFS close, and later re-open at T2 without updating the checkpoint. In this case, isSeizable uses the stale openSince T0 and may allow immediate seizure even though the new RFS episode at T2 has not aged through grace.
