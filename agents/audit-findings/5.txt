Stale commitmentDeficitSince preserved in paused remove-liquidity reconcile causes premature seizure eligibility

A new paused-mode remove-liquidity path reconciles positions without running a commitment checkpoint, preserving an old commitmentDeficitSince across an inactive interval. When the position is later reactivated and remains underbacked, the preserved age allows immediate grace-bypass seizure.

This PR adds a paused-mode removal branch (CoreHook._afterRemoveLiquidity -> VTSOrchestrator.reconcileAfterPausedRemove -> VTSPositionLib.reconcileAfterPausedRemove) that mirrors Uniswap’s live removal (including to zero liquidity), reduces commitment maxima, and clamps settled, but never runs VTSCommitLib.checkpointWithCommitment. If a position was underbacked before removal, pa.commitmentDeficit and pa.commitmentDeficitSince remain stored while the position becomes inactive, and cannot be checkpointed away because checkpoint requires an active position. On later reactivation, VTSLifecycleLinkedLib.validateSeize recomputes the deficit with checkpointWithCommitment only when a stored deficit exists; when prev>0 and next>0, VTSCommitLib._writeCommitmentDeficitToken preserves the old commitmentDeficitSince. CheckpointLibrary.isSeizable then treats the new underbacked episode as already aged past unbackedCommitmentGraceBypassTime, enabling immediate or premature seizure once severity thresholds are met. Before this PR, removal during pause was blocked and non-seizing decreases required RFS-closed, so reaching this stale-age-across-inactive state was significantly harder. The PR newly enables an unwind-only path during pause that bypasses RFS-closed gating and skips commitment checkpointing at the inactive boundary.
