4.2.1. Bucket Accounting Desync On Exemption Transition
Unit Name: mint
Location: contracts/evm/src/LCC.sol, LiquidityCommitmentCertificate.mint function,
bucket accounting after _mint(to, amount) and Bounds.isExempt() check
Description: The mint function skips bucket accounting ( wrappedBalances,
marketDerivedBalances ) when the recipient to has an exempt bound level
( Bounds.isExempt(ILiquidityHub(hub).boundLevel(factory, to)) ). If an address transitions from
exempt to non-exempt between multiple mint calls, the tokens minted while the address
was exempt are not tracked in any bucket, but the tokens minted after becoming non-
exempt are tracked. This creates a state where wrappedBalances[to] + marketDerivedBalances[to] <
balanceOf(to) for a non-exempt address.
The balancesOf view function only falls back to treating all balance as wrapped when
balanceSum == 0 && fullBalance > 0 . Once any bucket tracking exists (bucket sum > 0), it
returns only the tracked amounts. This means the diﬀerence between balanceOf(to) and the
bucket sum becomes eﬀectively stuck/non-transferable, because the transfer functions
( _handleNonProtocolToProtocol, _handleProtocolToNonProtocol, _handleProtocolToProtocol ) enforce
balance checks against marketDerivedBalances[from] + wrappedBalances[from] , not balanceOf(from).
Concrete scenario: (1) Address X is exempt, (2) mint(X, 0, 100) — balanceOf=100,
buckets=0, (3) X becomes non-exempt, (4) mint(X, 50, 0) — balanceOf=150,
wrappedBalances=50, marketDerived=0, (5) X tries to transfer 150 tokens: revert with
InsufficientBalance(50, 150) . 100 tokens are permanently stuck.
While bound level changes are admin-controlled operations, the setBoundLevel function in
LiquidityHub explicitly does NOT enforce on-chain validation of accounting consistency (as
noted in its comment: "We intentionally do not enforce this on-chain"), making this a viable
path to permanent fund loss if admin procedures are not followed perfectly.
User Verdict: True Positive
