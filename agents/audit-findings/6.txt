4.2.2. Balance Accounting Inconsistency
Unit Name: balancesOf
Location: contracts/evm/src/LCC.sol, LiquidityCommitmentCertificate.balancesOf function
Description: The balancesOf function can return values whose sum does not equal
balanceOf(account) , violating the stated invariant "balancesOf query function must return
results consistent with individual balanceOf queries."
The fallback condition (balanceSum == 0 && fullBalance > 0) only triggers when bucket tracking
is completely absent ( wrappedBalances[account] + marketDerivedBalances[account] == 0 ). If an account
has partial bucket tracking (e.g., balanceSum > 0 but balanceSum < fullBalance ), the function falls
through to return (wrappedBalances[account], marketDerivedBalances[account]) where the returned
sum is strictly less than balanceOf(account).
This state can arise when:
1. An account's bound level transitions from exempt (≥ BOUND_EXEMPT ) to non-exempt
(< BOUND_EXEMPT ). While exempt, mints and incoming transfers skip bucket
bookkeeping. After the transition, if the account then receives any bucket-tracked
transfer (adding a small amount to buckets), balanceSum > 0 causes the fallback to
no longer trigger. The function then returns the small bucket values, not the full
ERC20 balance.
2. Concretely: Account receives 100 LCC while exempt (buckets = 0, ERC20 = 100).
Admin removes exemption via setBoundLevel . An exempt protocol address transfers
10 LCC to the account via _handleProtocolToNonProtocol , which credits
marketDerivedBalances[account] += 10 . Now balanceSum = 10 , fullBalance = 110. balancesOf
returns (0, 10) — sum is 10, but actual ERC20 balance is 110.
Downstream impact: processSettlementLogic calls (, holderBal) = balancesOf(lcc, recipient) to
determine the settleable market-derived amount. With underreported holderBal , settlement
is capped at the bucket value rather than the true token holding, potentially locking
settlement entitlements. Additionally, the user's _handleNonProtocolToProtocol checks
totalBalance = marketDerivedBalances[from] + wrappedBalances[from] and reverts with
InsufficientBalance if the bucket sum is less than the transfer amount — eﬀectively locking
the 100 untracked tokens permanently.
User Verdict: True Positive
