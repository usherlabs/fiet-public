4.2.3. Exempt Status Transition Bucket Desync
Unit Name: balancesOf
Location: contracts/evm/src/LCC.sol, LiquidityCommitmentCertificate.balancesOf function,
interaction with setBoundLevel in BoundRegistry.sol
Description: When an account's bound level is changed from exempt (≥ non-exempt via setBoundLevel , a permanent bucket accounting desync can occur that
cannot be corrected through any on-chain mechanism. This desync causes BOUND_EXEMPT ) to
balancesOf to
return incorrect values and can lock user funds.
The attack/failure scenario:
1. Account receives LCC while exempt (e.g., during protocol operations). mint() skips
bucket updates due to if (Bounds.isExempt(...)) return; . ERC20 balance = 100,
buckets = (0, 0).
2. Admin calls setBoundLevel(account, BOUND_ENDPOINT) (level 1, non-exempt).
3. Account currently has balanceSum == 0 && fullBalance > 0 , so balancesOf returns (100,
0) via the fallback. So far, no apparent issue.
4. An exempt protocol address transfers 1 LCC to the account.
_handleProtocolToNonProtocol with exempt sender credits marketDerivedBalances[account] +=
1 . Now ERC20 balance = 101, buckets = (0, 1).
5. balancesOf now returns (0, 1) because balanceSum = 1 > 0 and account is not
exempt. The sum (1) does not match balanceOf(account) (101). 100 LCC are
"invisible" to the bucket system.
6. The account cannot transfer more than 1 LCC because _handleNonProtocolToProtocol
checks totalBalance = marketDerivedBalances[from] + wrappedBalances[from] (= 1) and
reverts with InsufficientBalance for any amount > 1, despite holding 101 LCC in
ERC20 balance.
The setBoundLevel function explicitly acknowledges this risk ("callers must not move who
into an exempt role while that address owns queued settlements") but does not enforce it
on-chain, and there is no mechanism to reconcile stale bucket state after a bound level
transition. The 100 untracked LCC tokens are eﬀectively permanently locked.
User Verdict: True Positive
