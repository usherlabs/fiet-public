# Summary Amendments for audit-findings/summary.txt

## Overview
This document records discrepancies identified between `agents/audit-findings/summary.txt` (generated by BugPoCer on 26 March 2026) and the authoritative protocol design documents:
- `contracts/evm/INVARIANTS.md` (primary source of truth for protocol invariants)
- Various documents under `agents/spec/` (especially `LiquidityHub.md`, `Value-to-Signal Model.md`, `Liquidity Commitment Certificates (LCCs).md`, `MEV-and-Adversarial-Ordering.md`, `Currency-Delta-Accounting.md`, `Tick-Indexed-Growth-Accounting.md`)

Priority of authority for this review is: implemented code first, `INVARIANTS.md` second, and `agents/spec/` documents as supporting context where they are still current.

The summary is largely aligned on core economic invariants (LCC backing domains, protocol-bounded LCC transfers, deficit separation, settle-growths-before-modify, single-curve execution). However, the following areas contain incorrect, incomplete, or self-contradictory statements.

## 1. Self-Contradictory Governance and Admin Model (Critical)
The summary contains two incompatible descriptions of administrative powers:

**Early section (lines ~94-107):**
- "Only the protocol owner (Ownable.owner()) can call pause/unpause functions on PausableVTS"
- "Only the VRLSignalManager owner can call setVerifier() to change the signal verification contract"
- Lists `prepareSettle()` under factory-only surfaces

**Later security assumptions section (lines ~327-344):**
- Claims GlobalConfig owner operations (including proxyCall) are "gated by multisig approval and timelock delay"
- Claims VRLSignalManager verifier changes are subject to "24â€“72 hour timelock delay"
- Claims "DAO or multisig can override owner and unpause in emergency scenarios"

**Reality per code and INVARIANTS.md:**
- `PausableVTS.sol:96,107,118`: `pausePool`, `unpausePool`, `setGlobalPause` are `onlyOwner`
- `VRLSignalManager.sol:73`: `setVerifier` is `onlyOwner`
- `GlobalConfig.sol:28`: `proxyCall` is `onlyOwner` with no timelock or multisig in the contract itself (timelock is off-chain deployment concern)
- No 24-72h timelock is implemented in the contracts for these operations.

This internal contradiction makes the threat model in the summary unreliable. The safe wording is: the contracts themselves do not enforce a timelock; any multisig or timelock protection would have to come from deployment and ownership configuration outside these contracts.

## 2. Incorrect Caller Classification for LiquidityHub.prepareSettle()
**Summary claims (line ~102-103):**
> Only registered factories (verified via LiquidityHub.isFactory()) can call initialize(), prepareSettle(), and createLCCPair() on LiquidityHub

The same mistake also appears later in the security assumptions section, where the Factory is described as the actor that calls `prepareSettle()`.

**Correct per code and INVARIANTS.md:**
- `createLCCPair` and `initialize` are factory-gated (`onlyFactory`).
- `prepareSettle`, `issue`, `cancel`, `confirmTake` etc. are **issuer-gated** (`_onlyIssuer` / `LiquidityHubLib.assertValidLcc`).

See:
- `contracts/evm/INVARIANTS.md:515-517` (MKT-04)
- `agents/spec/LiquidityHub.md:146` â€“ explicitly states MarketVaults call `prepareSettle`

This misclassification affects understanding of who can affect settlement accounting.

## 3. Incorrect Description of `settlePositionGrowths()` Access Surface
**Summary claims (lines ~348-350):**
> settlePositionGrowths can only be triggered by CoreHook or ProxyHook

**Reality in code:**
- `VTSOrchestrator.settlePositionGrowths(PositionId)` is itself a `public` function.
- When the position is valid and active, it can be called directly without an explicit caller-role modifier.
- During pause, the function has a narrower special case: growth-only settlement is allowed only from the canonical `CoreHook` for that pool.
- Separately, growth settlement is also reached from `checkpoint()`, MM settlement flows (`onMMSettle` / `VTSPositionLib.onMMSettle`), and liquidity modification paths.

This matters because the summary frames growth settlement as hook-gated, while the implemented surface is broader. That affects how readers reason about ordering, state realisation, and who may trigger accounting crystallisation.

See:
- `contracts/evm/src/VTSOrchestrator.sol:535`
- `contracts/evm/src/VTSOrchestrator.sol:886`
- `contracts/evm/src/VTSOrchestrator.sol:755`
- `agents/spec/MEV-and-Adversarial-Ordering.md:75-77`
- `agents/spec/Currency-Delta-Accounting.md:227`
- `contracts/evm/INVARIANTS.md:194-197`, `282-283`

## 4. Inconsistent Core vs Proxy Pool Terminology
**INVARIANTS.md (lines 14-15) and summary (lines 50-52) agree:**
- Core pool = LCC currencies
- Proxy pool = underlying currencies

**But `agents/spec/LiquidityHub.md` reverses it:**
- "ProxyHooks manage LCC-based proxy pools, proxying swaps to underlying core pools"
- "CoreHooks manage underlying-asset core pools"

This drift between spec documents should be cleaned up. On this specific point, the summary follows `INVARIANTS.md` and current protocol terminology more closely than `agents/spec/LiquidityHub.md`.

## 5. Proxy Swap Exact-Output / Fallback Behaviour
**Summary and INVARIANTS.md (MKT-05) state:**
- Proxy exact-output must revert when immediate vault liquidity unavailable.
- Queued-deficit exact-output on proxy pool is not supported.

**Older spec documents describe a different model:**
- `agents/spec/Liquidity Commitment Certificates (LCCs).md:52-54` and `Value-to-Signal Model.md:69` describe recipient parameter allowing "full order" with excess LCCs that are later settled.

The current invariants represent the stricter, safer design. The summary correctly reflects the implemented invariants here, but it should note that some older spec files describe an earlier or superseded proxy fallback model.

## 6. Minor Issues
- The summary notes "The target repository failed to compile with forge build" â€“ this may have affected PoC reliability.
- Some listed invariants in the summary (e.g. LCC totalSupply == sum of balances) are standard ERC20 properties but the protocol has additional bucket accounting invariants that are more important.
- The "Design Goals" section in the summary is mostly accurate but should explicitly reference `INVARIANTS.md` as the canonical source for current enforced behaviour.
- The summary should avoid presenting deployment assumptions (multisig, timelock, DAO override) as protocol-enforced facts unless they are backed by code or by environment-specific deployment documentation.

## Recommendations
1. Resolve the governance section to match the actual `onlyOwner` + off-chain timelock deployment model.
2. Correct the caller for `prepareSettle()` to "issuer-governed (for example, an authorised MarketVault), not factory-gated".
3. Replace the `settlePositionGrowths()` description with wording that matches the code: public callable for active positions, with additional internal use from checkpoint, MM settlement, and liquidity modification flows.
4. Harmonise terminology across all spec documents (core = LCC pool, proxy = underlying pool).
5. Mark outdated spec files (especially those describing relaxed proxy exact-output behaviour) as deprecated or superseded by the current invariants.

This amendments document should be read alongside `summary.txt` and `contracts/evm/INVARIANTS.md`.
