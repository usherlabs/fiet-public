1. About
This report was generated by Olympix's BugPoCer tool. BugPoCer automates end-to-end
vulnerability exploitation for smart contracts - moving beyond static warning to deliver
concrete, verifiable Proof-of-Concept (PoC) exploits. Built on Olympix's proprietary
infrastructure and Intermediate Representation (IR), BugPoCer combines results from
multiple analysis engines to identify confirmed security issues. Using an agentic AI
framework to synthesize insights across the Olympix toolchain, it enables teams to
instantly validate vulnerabilities, measure real-world impact, and confirm that patches are
eﬀective.
1.1. Methodology
BugPoCer breaks down a codebase into source units (function, contract, or library level).
While findings will only be searched for in units within the specified scope, BugPoCer
explores all relevant paths within the codebase. It couples this context with an extensive
knowledge store to identify potential issues. For each issue, BugPoCer automatically
generates a ready-to-run Foundry-format PoC test that reproduces the exploit under
realistic attack conditions.
Each finding in the Scan Report includes:
Vulnerability Details — Description, severity, aﬀected lines, and potential
impact
Exploit demonstration — A PoC test that reliably triggers the issue
The findings within this report are broken down into 3 categories:
1. True Positive (TP): Vulnerabilities that were proven true via failing PoC test
2. Unverified: Vulnerabilities for which BugPoCer was unable to write a compiling
PoC test
3. False Positive (FP): Vulnerabilities that were proven false via passing PoC test
Findings marked true positive should be taken into careful consideration. Unverified
findings should be reviewed as potential exploits. BugPoCer, as any human auditor, can
make mistakes. Therefore, false positives may be allocated some review time as a sanity
check.

1.3. Project Overview
Type: DeFi AMM Protocol (Uniswap V4 Hooks)
Description: Fiet Protocol is a DeFi protocol combining automated market making with
verified reserve liquidity (VRL), implemented as a dual Uniswap V4 hook system (CoreHook
for accounting/fee-sharing on LCC synthetic pools, ProxyHook for swap routing on
underlying asset pools), with Liquidity Commitment Certificates (LCC) as protocol-bound
ERC20 tokens representing claims across multiple liquidity domains, and a Value-to-Signal
(VTS) accounting and settlement system that manages commitment-backed liquidity
positions.
Design Goals:
Invariant safety over universal liveness (fail closed): The protocol deliberately
prefers invariant preservation and solvency-safe behaviour over always-on
execution, reverting when settlement cannot be completed safely rather than
under-settling
Single-curve execution model: All price discovery happens on the core LCC pool;
proxy pool swaps are fully neutralised at the Uniswap layer and must not advance
the proxy pool's own price state (MKT-05)
Domain-strict LCC backing: Every LCC mint must be attributable to exactly one
backing domain (wrapped, in-market, or signal-materialised), with no free minting
and strict supply accounting
Protocol-bounded transfer semantics: LCCs are market compatibility primitives,
not freely transferable assets; transfers require at least one protocol-bound
endpoint to prevent bypassing settlement/queue semantics
Strict execution-time validation for MM operations: Core protocol validates LCC
issuance against live pool state at execution time with no tolerance, preferring
client-side mitigation (7702 atomic re-validation) over relaxed on-chain checks
Conservative time-based enforcement: VTS timing parameters (grace periods,
seizure, expiry) must be configured conservatively per target chain's timestamp
behavior, with explicit risk decisions rather than silent defaults
Settle growths before liquidity mutations: Position growths must be settled before
any liquidity add/remove to prevent retroactive capture of previously-accrued
growth by newly-added liquidity
Key Patterns:
ERC20 - Liquidity Commitment Certificate (LCC)
ERC721 - Market Maker Position NFT
ResilientOracle Price Feed Integration
Uniswap V4 Hook Integration (ProxyHook + CoreHook)
Liquidity Hub with LCC Wrapping/Netting

LCC-02: For non-protocol to protocol transfers, queued-settlement ownership must
be annulled before bucket decrement to prevent 'bleeding' into the queue. LCC
bucket accounting must remain consistent with transfer flow.
MKT-05: Proxy swaps must be fully neutralised at the Uniswap layer so the proxy
pool does not advance its own price state. Any residual amountToSwap would
cause the proxy pool's own swap logic to execute, breaking this invariant.
Settle growths before liquidity mutations: Position growths must be settled (via
vtsOrchestrator.settlePositionGrowths) before any liquidity add or remove
operation to prevent retroactive capture of previously-accrued growth by newly-
added liquidity.
LCC mint is exclusively controlled by the Hub: LCC token minting is callable only
by the Hub (src/LCC.sol::mint is onlyHub). Issuer-only issuance/cancellation paths
are additionally guarded against invalid/uninitialised LCCs via
LiquidityHub._onlyIssuer which calls LiquidityHubLib.assertValidLcc.
Domain A wrapping conservation: When wrapping underlying to LCC via
LiquidityHub._wrap, directSupply[lcc] and reserveOfUnderlying[underlying] must
be incremented by exactly the amount of underlying actually received, and LCC
minted 1:1 against deposited underlying.
Domain conversion (wrapWith) must conserve value: wrapWith must conserve
value by converting one LCC claim into another without creating net backing; it
may reclassify between 'wrapped' and 'market-derived' buckets, but must not
mint value from nothing.
Proxy swap exact-output must revert when immediate underlying settlement
cannot be completed: When immediate output liquidity is unavailable from the
market vault, the proxy swap must revert rather than under-settling exact-output
or allowing the proxy pool AMM to execute.
VTS signal validity: A commit signal used for settlement must satisfy: commitId >
0, commit exists (expiresAt != 0), valid mmState.owner (non-zero), non-empty
reserves, and (for non-seizing operations) the signal must not be expired.
LCC issuer gating: Only authorized issuers can mint market-derived LCC (Domain
B). Issuer gating is enforced by LiquidityHub._onlyIssuer which validates both that
the LCC is valid and that the caller is on the issuer allowlist.
Deficit accounting separation: cumulativeDeficit (swap-driven outflow shortfall,
DICE principal for coverage-index attribution and fee-slash accounting) must be
tracked separately from commitmentDeficit (position-level insolvency gate for
RFS/seizability hardening). commitmentDeficit is not part of DICE principal
(totalDeficitPrincipal).
LiquidityHub lazy netting operations must be atomic: settle queue amounts,
nettedLCCsAsUnderlying, and directSupply must remain consistent across
concurrent operations on the same LCC pair
Checkpoint operations in VTS must validate that the checkpoint is in the correct
state (open) and that the PoolKey matches the PositionId's pool before any state
modification
LCC token burn operations must maintain consistency between burned amounts
and bucket accounting (direct vs market buckets)
